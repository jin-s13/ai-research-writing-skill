## NeurIPS 论文 Checklist

### 强制组成部分

所有 NeurIPS 投稿须包含已填写的 paper checklist。缺少该部分将面临**自动 desk rejection**。Checklist 出现在参考文献与补充材料之后，不计入页数限制。

### 16 项必填 Checklist

#### 1. Claims Alignment（主张一致性）
作者须核验摘要与引言中的主张与理论、实验结果一致，贡献、假设与局限表述清楚。

**检查项：**
- [ ] 摘要主张与实际结果一致
- [ ] 引言不过度 claim
- [ ] 贡献具体且可证伪

#### 2. Limitations Discussion（局限性讨论）
论文应含专门「Limitations」节，讨论强假设、对假设违背的鲁棒性、范围约束及影响性能的因素。

**应包含：**
- [ ] 专门 Limitations 节
- [ ] 诚实评估适用范围
- [ ] 方法可能失效的条件

#### 3. Theory & Proofs（理论与证明）
理论贡献须完整陈述假设，并提供完整证明（正文或附录，正文可有证明 sketch 以助理解）。

**检查项：**
- [ ] 所有假设正式陈述
- [ ] 提供完整证明（正文或附录）
- [ ] 正文有证明 sketch 以助直觉

#### 4. Reproducibility（可复现性）
作者须描述确保结果可验证的步骤：代码发布、详细说明、模型访问或适合贡献类型的 checkpoint。

**应提供：**
- [ ] 清晰可复现性声明
- [ ] 代码可得性信息
- [ ] 如适用，模型 checkpoint

#### 5. Data & Code Access（数据与代码访问）
应提供复现主要实验结果的说明（补充材料或 URL），含精确命令与环境规格。

**应包含：**
- [ ] 运行实验的精确命令
- [ ] 环境规格（requirements.txt、conda env）
- [ ] 数据访问说明

#### 6. Experimental Details（实验细节）
须说明训练细节：数据划分、超参及选择方法（正文或补充材料）。

**应记录：**
- [ ] 训练/验证/测试划分细节
- [ ] 所用全部超参
- [ ] 超参选择方法

#### 7. Statistical Significance（统计显著性）
结果须含误差条、置信区间或统计检验，并清楚说明计算方法与假设。

**应包含：**
- [ ] 误差条或置信区间
- [ ] 运行次数/种子数
- [ ] 计算方法（std dev vs std error）

#### 8. Compute Resources（算力资源）
须说明：计算节点类型（CPU/GPU）、内存、存储、单次运行时间、项目总算力。

**应记录：**
- [ ] GPU 类型与数量
- [ ] 单次训练时间
- [ ] 项目总算力

#### 9. Ethics Code Compliance（伦理守则）
作者确认遵守 NeurIPS Code of Ethics，并注明必要偏离。

**应核验：**
- [ ] 已阅读 NeurIPS Code of Ethics
- [ ] 确认合规
- [ ] 如有偏离，附理由

#### 10. Broader Impacts（更广泛影响）
在适用时讨论潜在负面社会应用、公平性、隐私风险及可能缓解策略。

**应涉及：**
- [ ] 潜在负面应用
- [ ] 公平性考量
- [ ] 隐私影响
- [ ] 缓解策略

#### 11. Safeguards（ safeguards）
高风险模型（语言模型、互联网爬取数据集等）须有受控发布机制与使用指南。

**应考虑：**
- [ ] 敏感模型发布策略
- [ ] 如需要，使用指南
- [ ] 如适当，访问控制

#### 12. License Respect（尊重许可）
所有既有资产须引用创作者、许可名称、URL、版本号并确认服务条款。

**应记录：**
- [ ] 数据集许可已引用
- [ ] 代码许可已遵守
- [ ] 含版本号

#### 13. Asset Documentation（资产文档）
新发布须用结构化模板记录训练细节、局限、同意程序与许可信息。

**对新数据集/模型：**
- [ ] Datasheet 或 model card
- [ ] 训练数据文档
- [ ] 已知局限

#### 14. Human Subjects（人类被试）
众包研究须含参与者说明、截图、报酬细节，并符合最低工资要求。

**应包含：**
- [ ] 任务说明
- [ ] 报酬细节
- [ ] 时间估计

#### 15. IRB Approvals（IRB 批准）
涉及人类被试的研究须有机构审查委员会或同等批准，并披露风险描述（投稿时匿名）。

**应核验：**
- [ ] 已取得 IRB 批准
- [ ] 完成风险评估
- [ ] 投稿时已匿名

#### 16. LLM Declaration（LLM 声明）
若将大语言模型作为核心方法论组成部分使用须披露；写作/编辑辅助不要求披露。

**应披露：**
- [ ] LLM 作为核心方法论组成部分
- [ ] LLM 如何使用
- [ ] （写作辅助不要求披露）

### 回答格式

作者对每题选「yes」「no」或「N/A」，可选 1–2 句理由。

**重要：** 审稿人被明确告知不得因诚实承认局限而惩罚。

## ICML 论文 Checklist

### Broader Impact Statement

ICML 要求在参考文献前文末附 Broader Impact Statement，**不计**入页数限制。

**必含要素：**
- 潜在正面影响
- 潜在负面影响
- 缓解策略
- 可能受影响的对象

### ICML 特定要求

#### 可复现性 Checklist

- [ ] 数据划分清楚说明
- [ ] 超参已列出
- [ ] 搜索范围已记录
- [ ] 选择方法已说明
- [ ] 算力资源已说明
- [ ] 代码可得性已声明

#### 统计报告

- [ ] 所有图含误差条
- [ ] 已说明 std dev 或 std error
- [ ] 已说明运行次数
- [ ] 比较方法时含显著性检验

#### 匿名化

- [ ] 正文无作者名
- [ ] 无致谢
- [ ] 无基金号
- [ ] 自引用第三人称
- [ ] 无可识别仓库 URL

## ICLR 要求

### LLM 披露政策（2026 年起）

ICLR 有专门 LLM 披露要求：

> "If LLMs played a significant role in research ideation and/or writing to the extent that they could be regarded as a contributor, authors must describe their precise role in a separate appendix section."

**须披露情形：**
- LLM 用于重要研究构思
- LLM 用于大量写作
- LLM 可被视为贡献者

**不要求披露：**
- 语法检查
- 轻微编辑辅助
- 代码补全工具

**未披露后果：**
- Desk rejection
- 发表后潜在问题

### ICLR 特定要求

#### 可复现性声明（可选但推荐）

声明中引用：
- 支撑材料
- 代码可得性
- 数据可得性
- 模型 checkpoint

#### 伦理声明（可选）

在 ≤1 页内讨论潜在关切，不计入页数。

#### Reciprocal Reviewing

- 3 篇及以上投稿的作者须为 ≥6 篇论文担任审稿人
- 每篇投稿须 ≥1 名作者注册并审 ≥3 篇

## ACL 要求

### Limitations 节（强制）

ACL 强制要求 Limitations 节：

**应包含：**
- 所作强假设
- 范围局限
- 方法可能失效情形
- 泛化关切

**重要：** Limitations 节**不计**入页数限制。

### ACL 特定 Checklist

#### Responsible NLP

- [ ] 已讨论偏见考量
- [ ] 如适用已评估公平性
- [ ] 已讨论双重用途关切

#### 多语言考量

若适用：
- [ ] 已涉及语言多样性
- [ ] 含非英语语言
- [ ] 已验证翻译质量

#### 人类评估

若适用：
- [ ] 已提供标注者细节
- [ ] 已报告一致性指标
- [ ] 已记录报酬

## OSDI 2026 要求

OSDI 关注操作系统设计与实现中的创新研究及量化/有洞见的经验。

### 投稿轨道

- **Research Track**：与往届 OSDI 类似，面向新颖系统研究
- **Operational Systems Track**（2026 年新增）：面向运营系统的设计、实现、分析与经验

### OSDI 投稿前清单

#### 格式
- [ ] ≤12 页（不含参考文献）
- [ ] 8.5"×11"、10pt/12pt 行距、双栏、Times Roman
- [ ] 正文块 7"×9"
- [ ] 已编页码
- [ ] 图表黑白可读
- [ ] 篇幅合适（勿注水；<6 页可能难以获充分考虑）

#### 内容
- [ ] 动机充分的重要问题
- [ ] 提出有趣、有说服力的方案
- [ ] 展示实用性与收益
- [ ] 结论恰当
- [ ] 清楚描述贡献
- [ ] 清楚说明相对先前工作的进展

#### 匿名化
- [ ] 双盲：无作者名、单位
- [ ] 系统/项目名已匿名（与 arXiv/报告不同）
- [ ] 未附 NDA 表格
- [ ] **Operational Systems 轨道例外**：可为上下文使用真实公司/系统名

#### 轨道相关
- [ ] 标题页与投稿表已标明轨道
- [ ] Operational Systems 论文：标题以「(Operational Systems)」结尾
- [ ] 每位作者最多 8 篇投稿

#### AI 政策
- [ ] 工作非全部或主要由 AI 生成（AI 编辑工具可接受）

## NSDI 2027 要求

NSDI 关注网络与分布式系统的设计原则、实现及实践评估。

### 投稿轨道

- **Traditional Research Track**：新颖想法 + 充分评估
- **Frontiers Track**（新增）：大胆想法，未必需完整评估
- **Operational Systems Track**：已部署系统及教训

### 预筛阶段

审稿人仅读 Introduction 以检查：

- [ ] 主题属于 NSDI 范围
- [ ] 表述可被 NSDI PC 成员理解
- [ ] Introduction 中阐明轨道标准
  - Research：新颖想法 + 评估证据
  - Frontiers：新颖非增量想法
  - Operational：部署场景、规模、社区教训

### NSDI 投稿前清单

#### 格式
- [ ] ≤12 页（不含参考文献），USENIX 格式
- [ ] 双栏、10pt、Times Roman
- [ ] 双盲匿名

#### 内容范围
- [ ] 贡献于网络系统设计
- [ ] 非超出范围主题（硬件体系结构、物理层、传感、UI）
- [ ] 标题页与投稿表已标明轨道

#### 重投规则
- [ ] 非上一 NSDI 截止被拒且无一 shot 修改机会
- [ ] One-shot 修订须：高亮修改 + 说明主要变更

## ASPLOS 2027 要求

ASPLOS 关注计算机体系结构、编程语言与操作系统的交叉。

### Rapid Review Round

**ASPLOS 独有且至关重要：**

- 审稿人**仅**读**前 2 页**
- 评估工作如何推进 Architecture/PL/OS 研究
- 多数投稿可能无法通过此阶段
- 缺合适审稿人的论文会提前退回

### ASPLOS 投稿前清单

#### 前 2 页（快速评审关键）
- [ ] 自洽：清楚陈述问题、方法与贡献
- [ ] 推进 Architecture、PL 和/或 OS 研究
- [ ] 非仅在其他领域使用 arch/PL/OS 的进展

#### 格式
- [ ] ACM SIGPLAN 格式（`\documentclass[sigplan,10pt]{acmart}`）
- [ ] ≤12 页（不含参考文献）
- [ ] 双盲匿名
- [ ] 提交文档无可识别信息

#### 投稿规则
- [ ] 每周期每位作者最多 4 篇
- [ ] 重投说明（如适用）描述变更
- [ ] 非上一 ASPLOS 周期立即重投
- [ ] 主题选择准确以利分配审稿人

## SOSP 2026 要求

SOSP 征集与计算机系统软件的设计、实现、分析、评估与部署相关的创新研究。

### SOSP 投稿前清单

#### 格式
- [ ] ACM SIGPLAN 格式（`\documentclass[sigplan,10pt]{acmart}`）
- [ ] ≤12 页技术内容（不含参考文献）
- [ ] A4 或 US letter，178×229mm 正文块
- [ ] 双栏、8mm 栏距、10pt/12pt 行距
- [ ] 已编页码，参考文献可超链接
- [ ] 图表无需放大可读，鼓励彩色但须灰度可读

#### 内容
- [ ] 动机充分的重要问题
- [ ] 提出并实现有说服力的方案
- [ ] 展示实用性与收益
- [ ] 清楚描述贡献与进展

#### 匿名化
- [ ] 双盲：用论文 ID 替代作者名
- [ ] 系统/项目名已匿名
- [ ] 自引用第三人称
- [ ] 无致谢或基金号

#### Artifact Evaluation（可选）
- [ ] 录用后计划提交 artifact
- [ ] 可复现材料已准备

#### Author Response
- [ ] 回复限于：纠正事实错误 + 回应审稿人问题
- [ ] 回复中无新实验或额外工作
- [ ] 建议控制在 500 词以内

## 系统会议通用要求

### 所有系统 Venue 共同期望

- [ ] **系统设计与实现** — 非仅算法
- [ ] **真实负载与评估** — 仅微基准不足
- [ ] **展示实际收益** — 延迟、吞吐、成本、能耗
- [ ] **与 SOTA 系统对比**
- [ ] **未同时投其他 venue**
- [ ] **允许先前 arXiv/技术报告**

### 每次投稿前

#### 论文内容

- [ ] 摘要 ≤ 词数限制（通常 250–300 词）
- [ ] 正文在页数限制内
- [ ] 参考文献完整且已核验
- [ ] 含 Limitations 节
- [ ] 所有图表有 caption
- [ ] Caption 自洽

#### 格式

- [ ] 使用正确模板（venue + 年份）
- [ ] 未改页边距
- [ ] 未改字号
- [ ] 满足双盲要求
- [ ] 页码（审稿版）或无页码（camera-ready）

#### 技术

- [ ] 所有 claim 有证据
- [ ] 含误差条
- [ ] 基线合适
- [ ] 超参已记录
- [ ] 算力已说明

#### 可复现性

- [ ] 代码将公开（或说明理由）
- [ ] 数据将公开（或说明理由）
- [ ] 环境已文档化
- [ ] 提供复现命令

#### 伦理

- [ ] 已考虑更广泛影响
- [ ] 局限诚实陈述
- [ ] 尊重许可
- [ ] 如需要已取得 IRB

#### 最终检查

- [ ] PDF 无编译错误
- [ ] 所有图正确渲染
- [ ] 所有引用可解析
- [ ] 补充材料已整理
- [ ] 会议 checklist 已完成
