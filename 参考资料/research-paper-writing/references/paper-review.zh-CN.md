# 论文审稿（Paper Review）

## 目标

用对抗式、审稿人风格的检查清单，在投稿前及早发现拒稿风险并修订论文。

## 核心原则

对论文质量追求极致：假定审稿人会追问每个薄弱点，并主动修复。

## 关键规则（不可违反）

每条主要主张，尤其在 Abstract 与 Introduction 中，必须：

1. 技术上正确，且
2. 有实验证据明确支撑。

若主张无支撑，则补充证据，或弱化/删除该主张。

## 论文通常因何被接收

1. 贡献充分（例如：新任务、新 pipeline、新模块、新设计选择、新实验发现或新洞见）。
2. 在公平对比下实证性能优于前人方法。
3. 对比实验与消融研究充分。

## 常见拒稿维度

| 拒稿维度 | 典型失败信号 |
| -------- | ------------ |
| 1. 贡献不足 | 1.1 针对的失败案例过于常见。<br />1.2 所提技术已被充分探索；预期收益可预测/众所周知。 |
| 2. 写作不清 | 2.1 缺少技术细节；工作不可复现。<br />2.2 某方法模块缺乏清晰动机。 |
| 3. 实证效果弱 | 3.1 相对前人方法的提升仅边际。<br />3.2 即使优于前人，绝对性能仍不够强。 |
| 4. 评估不完整 | 4.1 缺少消融研究。<br />4.2 缺少重要基线或重要评估指标。<br />4.3 数据集过于简单，无法证明方法真正有效。 |
| 5. 方法设计有问题 | 5.1 实验设定不现实。<br />5.2 方法有技术缺陷且显得不合理。<br />5.3 方法不鲁棒，需按场景调超参。<br />5.4 新设计引入的局限强于其收益，导致净收益为负。 |

## 稿末自审问题清单

修订时在稿末附近加入此清单。
投稿前用每个问题触发具体修改。

### 1. 贡献

1. 本文给读者什么新知识？
2. 我们是否解决真正有意义的失败案例，而非琐碎/常见案例？
3. 技术思路是否超越已充分探索的实践、真正非显而易见？
4. 收益是否令人惊讶或有洞见，而非可预测的改进？
5. 是否至少有一种清晰的新颖性类型（任务/pipeline/模块/设计发现/洞见）？

### 2. 写作清晰度

1. 有知识的读者能否根据论文复现方法？
2. 是否为每个关键模块提供足够技术细节？
3. 每个模块的动机是否明确且与挑战逻辑相连？
4. 术语与记号在各节间是否一致？
5. 每段是否只传达一个清晰信息且过渡顺畅？

### 3. 实验力度

1. 相对强基线的提升是否有意义，而非仅统计上微小？
2. 绝对性能是否达到目标会议/期刊的竞争水平？
3. 收益是否在多个数据集/设定/指标上一致？
4. 是否诚实报告优势与失败案例？

### 4. 评估完整性

1. 是否为所有关键设计选择做消融？
2. 是否在公平设定下包含所有强/近期基线？
3. 评估指标是否标准且对该任务充分？
4. 数据集/场景是否足够有挑战性以验证真实有效性？
5. 对比与消融协议是否记录清楚？

### 5. 方法设计合理性

1. 实验设定对实际使用是否现实？
2. 方法是否有隐藏技术缺陷或不合理假设？
3. 方法是否鲁棒，无需大量按例重调超参？
4. 收益是否大于新增复杂度与新局限？
5. 审稿人是否合理地认为净收益为负？

## 对抗式写作工作流

1. 以持怀疑态度的审稿人阅读论文。
2. 用论文中的明确证据回答上述每个问题。
3. 将每项标为 `pass`、`needs revision` 或 `needs new experiment`。
4. 据此修订主张、写作、实验或方法范围。
5. 重复直至无重大拒稿风险。
